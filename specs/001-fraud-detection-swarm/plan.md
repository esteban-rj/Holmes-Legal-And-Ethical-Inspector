# Implementation Plan: Healthcare Fraud Detection Swarm

**Branch**: `001-fraud-detection-swarm` | **Date**: 2026-06-24 | **Spec**: `specs/001-fraud-detection-swarm/spec.md`

## Summary

Build a decentralized "Swarm" system, based on the Blackboard (Pizarrón) pattern, for detecting fraud in public-health contracting (inspired by the Bogotá "Cartel de la Cardiología" case). Autonomous agents (Contracting, Logistics, Medical, Whistleblower) publish structured signals on a shared Blackboard; a Consensus Agent emits Critical Fraud Alerts **only** when signals originate from a user-initiated Investigation Request. The architecture is modular (SOLID), supports runtime registration of new agents, and runs as a single-process in-memory prototype for v1.

User-imposed tech constraints (binding):
- **RAG**: `langchain` is the abstraction for retrieval-augmented generation (used by Medical Agent and Whistleblower Agent).
- **Blackboard transport**: `asyncio.Queue` per channel/topic in v1 (in-memory; interface designed to be swapped for a broker).
- **LLM provider**: pluggable. First target model = `MiniMax-M3` (provider id `minimax`, model id `MiniMax-M3`). Provider selection lives in configuration.

## Technical Context

**Language/Version**: Python 3.11+

**Primary Dependencies**:
- `langchain` (core + community vectorstores) for RAG over tariff manuals / SOAT / ISS guidelines and over PQR text.
- `asyncio` (stdlib) — Blackboard transport is built on `asyncio.Queue`.
- `pydantic` v2 — signal schema validation and typed data model.
- `httpx` — outbound HTTP for allow-listed external calls (SECOP mock, OSRM mock).
- `structlog` — structured logging (per-agent, audit, network log).
- `typer` — CLI for submitting investigations and reading reports.
- `fastapi` + `uvicorn` — authenticated Investigation API.
- `pytest`, `pytest-asyncio` — unit + integration tests.

**LLM (configurable, default first try)**: `minimax/MiniMax-M3` (provider `minimax`, OpenAI-compatible chat endpoint). Provider/model are read from configuration; the LLM client is an interface (`LLMClient`) with one concrete adapter per provider. Swapping model = config change only.

**Storage**: In-memory for v1 (Blackboard store, alert store, investigation store behind interfaces so a relational store can be dropped in later). No DB in v1.

**Testing**: `pytest`, `pytest-asyncio`, `httpx.AsyncClient` for API integration tests, `mocker` style monkeypatching for outbound HTTP.

**Target Platform**: Linux server / macOS / Windows, Python 3.11+. Air-gapped deployment is a supported mode for Medical and Consensus Agents (and for Whistleblower when local LLM is configured).

**Project Type**: Library + CLI + thin HTTP API. The library (`holmes_swarm`) is the core deliverable; the CLI and API are thin adapters over it.

**Performance Goals**: Single-process prototype, prototype-scale volumes (thousands of records). Alert emission latency ≤ 60s p95 after a qualifying investigation-origin signal lands on the Blackboard. Investigation Report returned ≤ 5 minutes for ad-hoc runs (default timeout).

**Constraints**:
- FR-018 / FR-019: per-agent internet access enforced via allow-list; default for any new agent = no internet.
- FR-021: outbound HTTP must be logged without leaking sensitive payloads (PQR text, PHI).
- FR-022: Medical + Consensus must be air-gappable; Whistleblower air-gappable when local LLM is configured.
- Origin gating (FR-008 / FR-031–FR-034): `autonomous-monitoring` signals MUST NEVER trigger an alert.

**Scale/Scope**: 4 first-class detection agents (Contracting, Logistics, Medical, Whistleblower) + 1 consensus agent + Blackboard + Investigation API. Designed to grow by adding new agent classes via the standard contract (FR-010).
